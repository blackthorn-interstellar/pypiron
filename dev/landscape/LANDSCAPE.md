## How to read this

Each feature is rated:

| Symbol | Meaning |
|---|---|
| ✅ | Supported (verified or clearly documented) |
| 🟡 | Partial — limited, or with notable caveats |
| 🟦 | Planned / beta / on a public roadmap |
| ⬜ | Not supported |
| ❔ | Could not be determined |

**Coverage** is a single rollup across all 67 features (supported = 1, partial = ½). **Std** counts fully-supported items in the 8 PEP/protocol standards. **Stars** is the GitHub (or Codeberg) stargazers of the project's *own* source repo as a rough popularity proxy — blank for products whose only public repo is a client/CLI or that live off these hosts, since a CLI's stars aren't the product's. Treat ratings as a research snapshot, not a contract — verify against current docs before relying on any cell. See the per-product **Sources** for provenance.

## Product details

### PypIron

> An ultra-fast, no-database PyPI server in Rust for private registries — truth is immutable files on disk or S3, the PEP 503/691 index is a regenerable view, and one binary hosts, mirrors, and on-demand-proxies packages from a single URL.

- **Category:** Open source · hosting: self-hosted
- **Vendor:** Bryce Drennan (open-source project)
- **License:** MIT
- **Cost:** Free — MIT open source, one binary. A paid "Enterprise" tier (SSO/LDAP/SAML, audit log, fine-grained RBAC) is named on the roadmap but is not yet available; no pricing published.
- **Implementation:** Rust
- **First released / last updated:** 2025 / 2026-06-18 (latest commit; current tag v0.0.3)
- **Links:** [repo](https://github.com/brycedrennan/pypiron) · [homepage](https://github.com/brycedrennan/pypiron)
- **Popularity:** 0 stars · repo last pushed 2026-06-19
- **Coverage:** 71% overall · 7/8 standards · confidence: high
- **Performance:** README/BENCHMARK figures on AWS with the S3 backend: ~75,000 req/s on a 2-vCPU box (p99 2 ms) and ~440,000 req/s on 8 vCPU (p99 5 ms); publish→installable ~0.7 s (1 s with 10k packages hosted); 900 MB wheel upload in 15-20 s at ~50 MB RSS via streaming incremental hash (8 simultaneous large uploads without OOM); artifact download 3.9 Gbit/s (2 CPU) up to 48 Gbit/s (8 CPU), NIC-saturated. Brag targets in BENCHMARKS.md push hot read paths toward 80k-150k rps with p99 < 3 ms. Artifacts served immutable (cache-forever); indexes ETag-revalidated; S3 downloads can 302-redirect to presigned URLs so nodes never hold wheel bytes.
- **Scale / HA:** Stateless reads scale horizontally with zero coordination — point any number of nodes at one S3 bucket, no session affinity. Multi-node HA via a "sloppy" S3 leader lease (conditional writes, TTL, heartbeat, automatic failover); only the index writer is singular and split-brain merely duplicates idempotent rebuild work. Disk backend is explicitly single-node; multi-node requires S3. Designed and benchmarked against a full-PyPI-shaped corpus (~12-17M objects); a fingerprint audit makes steady-state cost scale with churn, not corpus size (~$0.25 of LISTs to audit a 17M-file tree). Cache-correct rather than cache-dependent, so one node is positioned as sufficient and an optional CDN is pure leverage.
- **Notable extras:**
  - On-demand PyPI proxy with origin-checked dependency-confusion defense — private names never fall through to upstream
  - Origin exclusivity: every package is exclusively 'private' or 'mirror', claimed at first write via a .origin marker; collisions are hard-rejected
  - --private-prefix namespace reservation with PEP 503 normalized-name matching (cf. PEP 752)
  - PEP 792 project-status (active/archived/quarantined/deprecated) relayed verbatim through sync and the proxy; first-party authoring not provided
  - Time-travel resolution preserved across mirrors: sync carries PyPI's true upload-time into sidecars so uv --exclude-newer / --exclude-older stay historically correct
  - Client-aware artifact delivery (auto/redirect/stream): presigned S3 302 for cache-safe clients (uv), byte-streaming for pip whose cache keys on URL
  - Synchronous upload mode (--sync-uploads) for publish-then-install CI read-your-writes
  - Self-healing crash-safe event markers (intent/commit pairs) + fingerprint audit; pypiron verify (read-only diff) and pypiron resync (rebuild-the-world)
  - Per-project download/usage attribution via Gmail-style username subaddressing into Prometheus labels
  - Sync filters (wheels/sdists, python/abi/platform tag globs, date windows) plus pypiron.toml config layering; HTTP-push mirror mode (the server is the single writer)
  - PEP 740 provenance relayed verbatim through sync and proxy (relay, never verified or minted; first-party attestation uploads refused)
  - No unsafe Rust (unsafe_code = forbid); no unwrap/panic on request or worker paths
  - Native TLS serving is not built in by design — production HTTPS is expected via a reverse proxy that terminates TLS (built-in TLS is a roadmap 'maybe'); outbound sync/proxy fetches use rustls
  - Denylist + mirror-freshness reconcile (airgap-friendly off-switch, OSV malicious-feed ingestion) is a documented design spec (docs/DENYLIST.md) — not yet implemented
- **Sources:** [https://github.com/brycedrennan/pypiron](https://github.com/brycedrennan/pypiron) · [https://github.com/brycedrennan/pypiron/blob/master/docs/VISION.md](https://github.com/brycedrennan/pypiron/blob/master/docs/VISION.md) · [https://github.com/brycedrennan/pypiron/blob/master/docs/DESIGN.md](https://github.com/brycedrennan/pypiron/blob/master/docs/DESIGN.md) · [https://github.com/brycedrennan/pypiron/blob/master/docs/STANDARDS.md](https://github.com/brycedrennan/pypiron/blob/master/docs/STANDARDS.md) · [https://github.com/brycedrennan/pypiron/blob/master/docs/ROADMAP.md](https://github.com/brycedrennan/pypiron/blob/master/docs/ROADMAP.md) · [https://github.com/brycedrennan/pypiron/blob/master/docs/CONFIGURATION.md](https://github.com/brycedrennan/pypiron/blob/master/docs/CONFIGURATION.md) · [https://github.com/brycedrennan/pypiron/blob/master/docs/COMPATIBILITY.md](https://github.com/brycedrennan/pypiron/blob/master/docs/COMPATIBILITY.md) · [https://github.com/brycedrennan/pypiron/blob/master/docs/TESTING.md](https://github.com/brycedrennan/pypiron/blob/master/docs/TESTING.md)

### devpi

> Self-hosted, MIT-licensed Python package server (server + web + client) providing on-demand PyPI caching mirrors, private staging indexes with inheritance, and primary/replica replication for HA.

- **Category:** Open source · hosting: self-hosted
- **Vendor:** devpi project (created by Holger Krekel; maintained by Florian Schulze / pyfidelity and community)
- **License:** MIT
- **Cost:** Free / open source (MIT). No paid tiers from the project; commercial support is third-party.
- **Implementation:** Python
- **First released / last updated:** 2013 / 2026-06-11 (devpi-server 6.20.2)
- **Links:** [repo](https://github.com/devpi/devpi) · [homepage](https://www.devpi.net/ (docs: https://doc.devpi.net/))
- **Popularity:** 1,191 stars · repo last pushed 2026-06-17
- **Coverage:** 60% overall · 3/8 standards · confidence: high
- **Performance:** On-demand caching mirror: a file is fetched from PyPI on first request, then served from local cache (works offline; re-checks upstream every ~30 min via mirror_cache_expiry). Default SQLite KeyFS allows one writer + many concurrent readers; the devpi-postgresql plugin removes the single-writer limit for multi-writer throughput. Multi-process serving is supported by running extra `--requests-only` worker processes behind nginx, with `/+status` routed to the primary. No official published throughput/latency benchmarks.
- **Scale / HA:** Primary/replica replication (formerly master/replica; now `--role=primary` / `--primary-url`) gives real-time read replicas for read scaling, geo-distribution, and failover; replicas serve the full API and relay writes to the primary. Failover is manual (a replica is manually promoted). Single logical writer (primary), so it is not write-horizontally-scalable; PostgreSQL backend supports multiple writers. Known to run at organization scale; no formal large-scale benchmark published.
- **Notable extras:**
  - Index inheritance / bases: an index can inherit from one or more base indexes (including root/pypi), producing a merged/virtual view
  - Per-user dev/staging indexes with an explicit push-to-production workflow (`devpi push`) for promotion between indexes
  - tox integration: the server can record and display tox test results per uploaded release file (acl_toxresult_upload)
  - devpi-pr plugin adds a package pull-request / review-and-merge staging workflow
  - devpi-tokens plugin adds macaroon-based, derivable, permission-restricted scoped tokens (can be narrowed client-side without contacting the server)
  - devpi-ldap plugin for LDAP/AD auth with group-to-ACL mapping; devpi-lockdown plugin to require authentication for read access
  - Pluggable PostgreSQL storage backend (devpi-postgresql, pg8000) stores both metadata and package files in the database and enables multiple writers; the only alternative backends are filesystem (SQLite KeyFS, default) and PostgreSQL — no S3/GCS/Azure object-storage backend exists
  - Default storage is an embedded SQLite KeyFS in the serverdir, so a standalone server needs no external database; the optional devpi-postgresql backend is an external database (hence no_database = partial)
  - Export/import of full server state to migrate across major devpi versions
  - PEP 691 JSON simple API served to installers (and used when fetching from mirrors) since devpi-server 6.6.0; the simple API declares meta api-version "1.0", so there are no PEP 700 fields (versions/size/upload-time) and no PEP 629 `pypi:repository-version` HTML marker (verified in server/devpi_server/views.py). Installers resolve and install fine, but `uv --exclude-newer` has no upload-time data to filter on (client_uv = partial)
  - Experimental core-metadata (PEP 658/714) support added in 6.20.0 via `--enable-core-metadata` and `mirror_provides_core_metadata`; the JSON simple files include an optional `core-metadata` field
  - PEP 592 data-yanked attribute is honored/passed through for mirror indexes (since 6.0.0) and exposed as the JSON `yanked` field; there is no first-class command to yank releases on a private index (delete requires temporarily setting the index volatile), so pep592_yank / delete_yank_mgmt are partial
  - Configuration via CLI flags, a YAML config file, or DEVPISERVER_* environment variables (e.g. --restrict-modify -> DEVPISERVER_RESTRICT_MODIFY); CLI > env > config-file precedence
  - No official Docker image or Helm chart from the project; community images (e.g. scrapinghub/devpi, jonasal/devpi-server) and a community Helm chart (topiaruss/helm-devpi) exist

### simpleindex

> A ~130-line PEP 503 simple-index server that maps package-name patterns to local directories or upstream 302 redirects via a TOML routing config — a minimal personal/private index, not a full repository manager.

- **Category:** Open source · hosting: self-hosted
- **Vendor:** Tzu-ping Chung (uranusjr)
- **License:** MIT
- **Cost:** Free (open source, MIT)
- **Implementation:** Python
- **First released / last updated:** 2020 / 2024-03-27

- **Popularity:** 56 stars · repo last pushed 2024-03-27
- **Coverage:** 25% overall · 1/8 standards · confidence: high
- **Performance:** Tiny Starlette + uvicorn ASGI app (single process by default). Local file (path) routes read the entire distribution into memory with path.read_bytes() per request — no streaming, no Range, no caching. HTTP routes are a single 302 redirect to upstream, so the client downloads directly from PyPI/upstream and simpleindex itself does no byte proxying. No published benchmarks; intended for small/personal indexes.
- **Scale / HA:** No built-in HA, clustering, or multi-node support. The server is effectively stateless (it reads files from disk or emits redirects), so multiple instances could in principle sit behind a load balancer, but nothing in the project supports or documents this. Designed for a single small private/personal index, not large-scale operation.
- **Notable extras:**
  - Core idea is a 'virtual index from routing rules': TOML [routes."pattern"] blocks map package-name patterns (with {var} substitution) to either a local directory (source = "path") or an upstream URL (source = "http"); first matching route wins
  - Pluggable custom route types via the 'simpleindex.routes' entry-point group — subclass simpleindex.routes.Route and implement async get_page()/get_file(); README ships an Amazon S3 source as the documented example, installed via 'pipx inject'
  - HTTP routes are implemented purely as 302 redirects to the upstream simple index (e.g. https://pypi.org/simple/{project}/) — no caching, mirroring, or byte proxying; the client follows the redirect and downloads from upstream
  - Config file supports a 'path.toml::key.path' selector so the configuration can be embedded under a key inside another TOML file (e.g. pyproject.toml)
  - The [server] section is passed straight through as kwargs to uvicorn.run(), so any uvicorn option (host, port, workers, ssl_keyfile/ssl_certfile) can be set — undocumented but functional, including TLS
  - Read-only by design: package files are placed on disk manually; there is no upload, deletion, yank, or management API
  - Path routes only list filenames that parse as valid wheels or sdists via packaging.utils; anything else (incl. eggs) is silently ignored
  - Generated HTML is bare PEP 503 anchors only (<a href="./{name}">{name}</a>) — no data-requires-python, no #sha256= hashes, no data-yanked, no repository-version meta tag

  - delete subcommand removes packages; verify subcommand repairs missing files and garbage-collects unowned files
  - diff-file generation lists new/changed files for air-gapped, two-stage offline syncs
  - keep_index_versions retains timestamped prior index files as a rollback safeguard
  - Mirrors legacy /pypi/<pkg>/json metadata (json=true) alongside the simple index, enabling a statically-served legacy JSON API
  - Generates PEP 503 HTML and PEP 691/700 JSON simple indexes simultaneously (simple-format = ALL); JSON includes api-version 1.1 with versions, size, and upload-time fields, and data-yanked (PEP 592) is emitted in both formats

  - Official Docker image (pypa/bandersnatch) bundling runner.py for scheduled periodic syncs
  - Cross-platform: Linux, macOS, Windows (Windows requires symlink/elevation privileges)
  - Per-file SHA256 verification with selectable compare-method (hash vs stat) and digest_name; can consume any PEP 691-advertised digest

- **Scale / HA:** No self-managed nodes — availability/scaling are inherited from GitHub Pages (CDN-backed, effectively HA for reads, stateless). Practical limits are GitHub Pages soft limits (~1 GB site size, ~100 GB/month bandwidth, ~10 builds/hour) and the practicality of maintaining many packages as static HTML committed via PRs.   - PEP 503 is only partial: the page structure is simple-index-shaped, but the per-file hrefs are git+https VCS links with #egg= fragments and no hash, not the file-download URLs PEP 503 mandates — so it works only with pip's legacy egg-fragment/VCS handling and is pip-only in practice, not portable across installers
  - Client split is real and verified, not a generic install-clients-move-together case: pip installs via --extra-index-url, but spec-conformant clients that expect file links + a sha256/md5 hash reject the egg-fragment VCS anchors — uv fails outright (astral-sh/uv#2602: uv treats the #egg= fragment as a malformed hash), and Poetry/PDM/Hatch/Pipenv likewise do not consume egg-fragment VCS index links; twine has no upload target (static site, no upload API) and flit has no install capability
  - Installed exactly like a real index: pip install <pkg> --extra-index-url https://<owner>.github.io/<repo>/ — pip queries the static index and resolves the per-package page to its git+https VCS link (no manual copy-paste)
  - Packages are registered/updated/deleted through manually-triggered GitHub Actions workflows that open a PR; nothing is published until the PR is merged (PR acts as a review/staging gate, and Git/PR history is the change audit trail)
  - Privacy is delegated to GitHub: the index and its package names/versions are public, but linked code lives in private GitHub repos that pip can only clone with GitHub credentials (.netrc login/PAT), e.g. in Docker via a mounted secret
  - Dependency-confusion awareness (not enforcement): each package page queries the public PyPI JSON API client-side and, if a same-named package with a higher version exists on PyPI, replaces the shown install command with a warning

### pep381client

> The original (2010) PEP 381 PyPI mirroring client by Martin v. Löwis - a cron-driven CLI that bulk-mirrors all of PyPI to local disk for serving by a separate static web server; superseded by bandersnatch and now defunct (PEP 381 itself is Withdrawn).

- **Category:** Defunct · hosting: self-hosted
- **Vendor:** Martin v. Löwis (individual author)
- **License:** AFL-3.0 (Academic Free License v3.0)
- **Cost:** Free / open source
- **Implementation:** Python (Python 2.x)
- **First released / last updated:** 2010 / 2011-05-09 (v1.5)
- **Links:** [repo](https://bitbucket.org/loewis/pep381client) · [homepage](http://bitbucket.org/loewis/pep381client/)
- **Coverage:** 16% overall · 0/8 standards · confidence: medium
- **Performance:** Cron-driven incremental sync (recommended every 15 min). Efficiency comes from PyPI's changelog() XML-RPC serial cursor plus ETag caching: only changed /simple/ pages and package files are re-fetched, so steady-state runs are cheap. No published benchmarks; a full PyPI mirror in 2010-2011 was on the order of tens of GB. Single-threaded synchronous downloads (no concurrency), which is part of why the more robust/concurrent bandersnatch superseded it.
- **Scale / HA:** Single-process cron job, no HA, clustering, or multi-node coordination. Resumable after interruption via the persisted changelog serial cursor (last_completed), so a killed run continues where it left off. The output is a static file tree (web/simple, web/packages, web/serversig, web/local-stats, last-modified) served by a separate web server (Apache/nginx), which scales independently of the sync tool.
- **Notable extras:**
  - Direct code ancestor of bandersnatch - the bandersnatch README's Kudos section states 'This client is based on the original pep381client by Martin v. Loewis.'
  - Mirror-only tool: produces a static PEP 381 file tree (web/simple, web/packages, web/serversig, web/local-stats/days, last-modified) and leaves HTTP serving to a separate web server (Apache/nginx) - it is not itself a server, has no HTTP listener, auth, or upload
  - Incremental sync via PyPI's changelog() XML-RPC serial cursor; persists progress so it is resumable after interruption
  - Mirrors PEP 381 serversig/ DSA index signatures verbatim so clients can verify the simple pages - but the tool does not generate or verify any signatures itself
  - Format-agnostic file mirroring: copies whatever files PyPI lists under /packages/ (sdists, eggs, and later wheels) without inspecting type
  - Automatically deletes projects/files removed upstream (404/301) to keep the mirror in sync - but offers no operator-facing retention policy, delete, or yank management
  - Zero third-party dependencies (pure Python 2 stdlib); run via the pep381run script under cron, e.g. */15 * * * * pep381run -q /var/pypi
  - Unmaintained since v1.5 (2011-05-09); only 4 releases (v1.2-v1.5, 2010-2011); original Bitbucket Mercurial repo effectively offline since Bitbucket dropped Mercurial in 2020
  - Underlying PEP 381 mirroring protocol was Withdrawn (PyPI moved behind a Fastly CDN in May 2013), making the dedicated mirror client largely obsolete
  - Licensed under the Academic Free License v3.0 per PyPI metadata ('License: Academic Free License, version 3'); no SPDX license classifier is set on the package
  - Source inspection of a close fork shows downloads are written directly to the target path (open+write, no tmp-then-rename) and no md5/sha hash verification of downloaded files

### proxypypi

> A tiny, database-free, on-demand caching/pull-through PyPI proxy from 2013 that lazily fetches and rewrites package files to local references and can also serve private packages dropped into its cache directory.

- **Category:** Defunct · hosting: self-hosted
- **Vendor:** Richard Jones (individual maintainer; original author of the legacy PyPI/cheeseshop codebase)
- **License:** BSD
- **Cost:** Free (open source, BSD-licensed)
- **Implementation:** Python
- **First released / last updated:** 2013 / 2013-05-14
- **Links:** [repo](https://bitbucket.org/r1chardj0n3s/proxypypi) · [homepage](https://pypi.org/project/proxypypi/)
- **Coverage:** 16% overall · 1/8 standards · confidence: high
- **Performance:** No published benchmarks (throughput/latency/memory). Design goal is faster repeat installs via local caching after the first fetch. To survive pip's short connection timeout, it streams a background fetch of the upstream file to the client rather than downloading-then-serving; v1.2.2 removed the partial-download path for missing external files. Single synchronous process; not optimized for high concurrency.
- **Scale / HA:** Single-node, single-process tool backed by one local filesystem cache directory. No HA, clustering, multi-node, or stateless horizontal-scaling story. Optionally runs as a background daemon (PID/log/console-output files + start/stop control commands). Explicitly positioned as a simpler alternative to devpi-server, and the README itself recommends using devpi instead. Effectively abandoned since 2013; its Bitbucket Mercurial repo is gone (404 after Bitbucket's 2020 Mercurial sunset).
- **Notable extras:**
  - Streaming background fetch keeps pip's short-timeout connection alive while the upstream file is retrieved (its defining trick)
  - Rewrites BOTH PyPI-hosted and externally-hosted download URLs to local references, handling the externally-hosted-file era of pre-2014 PyPI
  - Drop-in private package hosting: just place distribution files in the cache root (package-named subdirs optional, not required)
  - Database-free by design (filesystem only) explicitly to be simpler to deploy than devpi-server
  - Background daemon mode with PID file (-P), log file (-l), console-output file (-o) and daemon start/stop control commands; cache dir set via -d
  - Self-deprecated: README warns 'Don't use this software. Use devpi instead!'
  - No cache upgrade/refresh of already-cached packages ('This isn't done yet') so cached versions are effectively pinned forever
- **Sources:** [https://pypi.org/project/proxypypi/](https://pypi.org/project/proxypypi/) · [https://pypi.org/project/proxypypi/1.2.2/](https://pypi.org/project/proxypypi/1.2.2/) · [https://pypi.org/pypi/proxypypi/json](https://pypi.org/pypi/proxypypi/json) · [https://bitbucket.org/r1chardj0n3s/proxypypi](https://bitbucket.org/r1chardj0n3s/proxypypi)

## pypiron vs. the superset

pypiron covers **71%** of the superset by design — it is a deliberately scoped static-file server, not an everything-registry. Features it does **not** implement (by category) are the explicit non-goals and the candidate roadmap:

- **Repository Roles:** Multiple repositories/indexes, Retention / cleanup / GC policies
- **Auth & Access Control:** Token / API key auth, Scoped / per-package tokens, OIDC Trusted Publishing, LDAP / Active Directory, SSO (SAML/OAuth/OIDC login), Per-repo / per-package permissions
- **Security & Supply Chain:** Vulnerability / CVE scanning, Malware scanning, License compliance / policy, SBOM generation / export, Audit logging, Quarantine / staging gates
- **Deployment & Operations:** Kubernetes / Helm chart, Project page / README rendering, Download statistics (per package/version)
